# Xanathar (XGE) — Plano de Implementação

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Trazer o pacote completo de jogador do Guia de Xanathar para Todas as Coisas (31 subclasses incl. Hexblade+CHA na arma de pacto, ~95 magias com mecânicas curadas, 15 talentos raciais com prereq de raça, ~14 invocações de Bruxo, itens mágicos comuns) pro sistema dnd5e, reutilizando a infra de fontes do Tasha.

**Architecture:** Nova fonte `xanathar` em `sources.js` + arquivos `xanathar-*.json` compostos no `SrdProvider` (união carimbada, gating no consumo). Código novo só em 3 pontos: prereq de raça nos FeatPickers, Hexblade no `WarlockPactPanel`, e lista de Clérigo pro Feiticeiro Alma Divina no `useClassSpells`. Todo o resto é dado extraído do PDF PT-BR do dono via esteira `scripts/xanathar/` (pymupdf).

**Tech Stack:** React+Vite, Vitest, Python+pymupdf (extração), JSON em `public/srd-data/`.

**Spec:** `docs/superpowers/specs/2026-07-07-xanathar-design.md`

```js
/**
 * Pré-requisitos de talento por RAÇA (talentos raciais do Xanathar).
 * O match aceita tanto o índice da raça quanto o da sub-raça da ficha —
 * "elfo-negro-drow" é sub-raça de "elfo", mas conta como raça pro prereq.
 */
const RACE_LABEL = {
  anao: 'Anão', elfo: 'Elfo', halfling: 'Halfling', humano: 'Humano',
  draconato: 'Draconato', gnomo: 'Gnomo', 'meio-elfo': 'Meio-Elfo',
  'meio-orc': 'Meio-Orc', tiefling: 'Tiefling',
  'alto-elfo': 'Alto Elfo', 'elfo-da-floresta': 'Elfo da Floresta',
  'elfo-negro-drow': 'Drow',
}

export function meetsRacePrereq(prereq, { race, subrace } = {}) {
  if (!prereq || prereq.type !== 'race') return true
  return (prereq.races ?? []).some(r => r === race || r === subrace)
}

export function formatRacePrereq(prereq) {
  return (prereq?.races ?? []).map(r => RACE_LABEL[r] ?? r).join(' ou ')
}
```

## Riscos e regras transversais (releia antes de cada fase)

1. **Bump `srd-data-vN` em todo merge que toca `public/srd-data`** — sem isso o SW serve dado velho em produção.
2. **Slug ausente no catálogo** (magia PHB sem tradução): omitir da tabela com comentário no padrão `GENIE_KIND.marid` — nunca inventar slug.
3. **Ids de choice devem bater com o PHB** (`patron`, `divine_domain`, `sacred_oath`, `ranger_archetype`, `druid_circle`, `bard_college`, `sorcerous_origin`, `martial_archetype`, `monastic_tradition`, `roguish_archetype`, `arcane_tradition`, `eldritch_invocations`) — o merge casa por `id`; id errado cria choice duplicada na UI. Conferir cada um em `phb-class-choices-pt.json` antes de estruturar.
4. **`alma-divina`**: o slug da option (Task 12) e o do gate no `useClassSpells` (Task 9) são o MESMO string.
5. **Formato parseável**: sem `Features ... por nível:` + `• Nv N —` no texto, a subclasse não ganha cards nem trackers. O teste de parse pega, mas escreva já no formato.
6. **Fidelidade ao PDF**: desc é transcrição; metadados (prereq, tabelas, hints) são interpretação SUA — é isso que o dono revisa.
